# github-flow-helper

Git/GitHubの標準開発ワークフローに沿って、ブランチ作成からプルリクエスト（PR）の作成、Issueとの連携、進捗管理を安全かつ効率的に自動化するためのスキルです。

## 1. Git運用ルール

### 禁止事項
* **mainブランチの保護**:
  - `main`ブランチへの直接の `git commit` および `git push` は絶対に行わないでください。
  - `main`ブランチへの直接のマージ（`git merge`）などのローカルでのマージ操作も禁止です。
* **マージの承認**:
  - 作業ブランチから `main` ブランチへのマージは、必ず GitHub 上のプルリクエスト（PR）を介し、ユーザー自身による承認・マージ実行のもとで行われなければなりません。AIエージェント自身が `main` へのマージを完了させてはなりません。

### 許可事項 (自動実行可能)
* **作業ブランチへの操作**:
  - `main` ブランチ以外の開発用・作業用ブランチにおける `git commit` および `git push` は、ユーザーの個別の許可を待たずに自動実行して構いません。
* **読み取り専用コマンド**:
  - リポジトリの状態を確認するため、以下の読み取り専用コマンドは自動実行できます: `git status`, `git log`, `git diff`, `git branch`

### コミットメッセージのルール
変更内容を簡潔に表す英語または日本語のメッセージを指定し、適切なプレフィックスを付与してください。
* `feat:` 新機能の実装や追加
* `fix:` バグ修正や挙動の調整
* `clean:` リファクタリングや不要になったコード/機能の削除
* `docs:` ドキュメントやルールファイルの作成・更新
* `test:` ユニットテストの追加・変更

## 2. GitHub Issue起票・管理ルール

### Issue起票ルール
* **言語**: Issue のタイトルおよび本文は、すべて日本語で記述してください。
* **自動化**: `gh` CLI を使用して、エンジニアの手間を省きつつ正確に起票します。
* **タイミング**: `implementation_plan.md` が承認された後、または新機能の実装を開始する直前に実行します。
* **一貫性**: 常に同じ構成で情報をまとめることで、後からのトレーサビリティを確保します。

### 起票手順
1. **プランの承認**: ユーザーから実装プランの承認を得る。
2. **内容の提示**: 起票する Issue のタイトルと本文（マークダウン）を作成し、ユーザーに確認を求める。
3. **起票の実行**: 承認された内容で `gh issue create` コマンドを実行する。
4. **タスクへの紐付け**: 作成された Issue 番号を `docs/progress/task.md` (またはそれに準ずる進捗管理ファイル) に追記する。

### 進捗管理
* **チェックボックスの更新**: 実装が進み、完了条件（Checklist）を満たした項目は、`gh issue edit --body` を使用して随時 `[x]` に更新してください。
* **報告の前提条件**: すべてのチェックボックスが `[x]` になっていることを確認してから、完了報告を行ってください。

### Issueフォーマット
* **タイトル**: `[prefix]: タイトル` (例: `feat: 学習結果のデータベース保存`)
* **本文構成**:
  - **目的**: 何を解決するためのタスクか。
  - **実装概要**: `implementation_plan.md` に基づく主要な変更点。
  - **完了条件 (Checklist)**: タスクの完了を判断する基準。

## 3. プルリクエスト (PR) 作成ルール

### 検証の義務
* PRの作成（およびリモートへのプッシュ）を行う前に、必ず検証用スクリプト `./scripts/verify.ps1` （またはプロジェクトに定義されたGradleタスク/npmスクリプト等）を実行し、すべて正常にパスすることを確認してください。

### PR作成手順
1. **カレントブランチ of 確認**: 現在 `main` ブランチにいる場合は、`main` から分岐した新しい作業ブランチ（例: `feat/xxx`, `fix/xxx`, `clean/xxx`）を作成して移動します。
2. **検証の実行**: `./scripts/verify.ps1` などの検証コマンドを実行し、エラーがないことを確認します。
3. **コミットとプッシュ**: 適切なプレフィックスを付けたメッセージでコミットし、リモートにプッシュします。
4. **プルリクエストの作成**: `gh pr create` コマンドを使用してPRを作成します。タイトル・本文は日本語。
5. **本文 (Body) 構成**:
   - 変更の目的（背景）
   - 主な変更内容（ファイルの箇条書きなど）
   - 関連する Issue があればそのリンク（例: `Closes #Issue番号`）
6. **完了報告**: 作成されたPRのURLをユーザーに報告します。

---

## 4. 完了報告ワークフロー
* タスク完了時は、`walkthrough.md` で行った変更の要約を作成し、対応する GitHub Issue に `gh issue comment <issue-number> --body-file walkthrough.md` （または要約内容）でコメントを投稿し、Issueをクローズするかユーザーに確認してください。
