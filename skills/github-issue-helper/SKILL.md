# github-issue-helper

GitHub Issueをタスク管理の起点として使用し、進捗状況の管理（チェックリストの更新）から、成果物の報告、Issueのクローズまでのライフサイクルを自動化・支援するためのスキルです。

## 1. GitHub Issue起票ルール

* **言語**: Issue のタイトルおよび本文は、すべて日本語で記述してください。
* **自動化**: `gh` CLI を使用して、正確かつ省力で起票します。
* **タイミング**: `implementation_plan.md` が承認された後、または新機能の実装を開始する直前に実行します。

### 起票手順
1. **プランの承認と情報確認**:
   - `implementation_plan.md` の内容から、以下の「必須項目」を埋めるための情報が十分に揃っているか確認します。
   - 情報が不足している場合は、**起票プロセスに進む前に必ずユーザーに質問し、情報を補完してください**。
2. **必須項目**:
   - **目的 / 背景**: なぜこのタスクが必要なのか、何を解決するのか。
   - **実装概要**: どのような変更を行うのか、具体的な設計や影響範囲。
   - **完了条件 (Checklist)**: 何をもってこのタスクが「完了」したとみなすか（テスト項目、確認項目を含む）。
3. **ドラフトの提示**: 必要な情報が揃ったら、Issueのタイトルと本文（マークダウン形式）の下書きを作成し、ユーザーに最終確認を求めます。
4. **起票の実行**: ユーザーの承認を得た後、`gh issue create` コマンドで起票します。
5. **タスクへの紐付け**: 作成された Issue 番号を `task.md` (またはそれに準ずる進捗管理ファイル) に追記します。

### Issueフォーマット
* **タイトル**: `[prefix]: タイトル` (例: `feat: 学習結果のデータベース保存`)
* **本文構成**:
  - **目的**: 何を解決するためのタスクか。
  - **実装概要**: `implementation_plan.md` に基づく主要な変更点。
  - **完了条件 (Checklist)**: タスクの完了を判断する基準（マークダウンのチェックボックス `- [ ]`）。

---

## 2. 進捗管理 (チェックリスト更新)

* **チェックボックスの更新**: 実装が進み、完了条件（Checklist）を満たした項目は、`gh issue edit <issue-number> --body` を使用して随時 `[x]` に更新してください。
* **報告の前提条件**: すべてのチェックボックスが `[x]` になっていることを確認してから、完了報告を行ってください。

---

## 3. 完了報告ワークフロー

* タスク完了時は、`walkthrough.md` で行った変更の要約を作成し、対応する GitHub Issue に `gh issue comment <issue-number> --body-file walkthrough.md` （または要約内容）でコメントを投稿します。
* コメント投稿後、ユーザーに確認のうえ `gh issue close <issue-number>` でクローズするか、ユーザー自身にクローズしてもらいます。
